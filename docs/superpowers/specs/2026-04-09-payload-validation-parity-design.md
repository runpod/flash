# AE-2744: Payload Validation Parity Between Dev Server and Deployed Endpoints

## Problem

The dev server (`flash run`) and deployed QB endpoints validate input payloads differently. A function with all-optional parameters (e.g., `async def run(msg=None)`) accepts `{"input": {}}` locally but silently times out at ~42 seconds when deployed. The RunPod platform does not dispatch jobs with empty or null `input` dicts to workers, but provides no error message -- the job sits in the queue until platform timeout.

This causes a "works locally, fails in production" trap for developers.

## Scope

- **In scope**: `flash` repo only. Dev server QB routes, deployed handler templates, `create_deployed_handler`.
- **Out of scope**: `flash-worker` (Live Serverless FunctionRequest path), LB endpoints (already validated by FastAPI at HTTP layer in both environments), SDK client-level validation (future work).

## Root Cause

### Dev server (`flash run`)

QB routes use Pydantic models generated by `make_wrapped_model(name, input=(inner_model, ...))`. FastAPI validates that `input` exists and matches the inner model schema. For functions with all-optional parameters, `{"input": {}}` passes Pydantic validation and the function executes with defaults.

**Relevant code**: `_run_server_helpers.py:call_with_body()`, `run.py:446-538`

### Deployed QB endpoints

Two code paths, neither validates empty input:

1. **Handler templates** (`handler_generator.py:36`): `job_input = job.get("input", {})` -- silent default to empty dict.
2. **`create_deployed_handler`** (`generic_handler.py:240-241`): Explicitly converts None to `{}` with no validation.

Neither path matters in practice because the RunPod platform never dispatches empty-input jobs to workers. But they provide no defense-in-depth.

## Design

### 1. Dev Server Validation

**File**: `src/runpod_flash/cli/commands/_run_server_helpers.py`
**Function**: `call_with_body()`

Add validation at the top of `call_with_body`. Use Pydantic's `model_fields_set` attribute to distinguish "user sent empty dict" from "user explicitly sent null values":

```python
async def call_with_body(func, body):
    if hasattr(body, "model_fields_set") and not body.model_fields_set:
        raise HTTPException(
            status_code=422,
            detail=(
                "Empty input: RunPod serverless requires at least one field "
                "in the input dict. Use explicit values or pass null for "
                'optional parameters, e.g. {"input": {"param_name": null}}.'
            ),
        )
    # ... existing logic unchanged
```

**Why `call_with_body`**: Single choke point for all QB handler execution when `needs_body=True`. Zero-param functions use `needs_body=False` and bypass this path entirely, so they are unaffected.

**Why `model_fields_set`**: Distinguishes `{"input": {}}` (empty set, all defaults) from `{"input": {"msg": null}}` (set contains "msg", user explicitly provided a value). This prevents false positives on intentional null values.

### 2. Deployed Handler Template Validation

**File**: `src/runpod_flash/cli/commands/build_utils/handler_generator.py`
**Templates**: `DEPLOYED_HANDLER_TEMPLATE`, `DEPLOYED_CLASS_HANDLER_TEMPLATE`

Add early rejection before function invocation:

```python
def handler(job):
    job_input = job.get("input") or {}
    if not job_input:
        return {
            "error": (
                "Empty or null input. RunPod serverless requires at least one "
                "field in the input dict. Use explicit values or pass null for "
                'optional parameters, e.g. {"input": {"param_name": null}}.'
            )
        }
    # ... existing function call logic
```

Same pattern for the class handler template.

**Why defense-in-depth**: The RunPod platform currently prevents empty-input jobs from reaching workers. This validation catches the case if platform behavior changes, or if the handler is tested directly (e.g., `python handler_<name>.py` with manual input).

**Error response shape**: Plain `{"error": ...}` dict, consistent with existing error returns in both templates (e.g., line 58 in function template, line 124-130 in class template).

### 3. `create_deployed_handler` Validation

**File**: `src/runpod_flash/runtime/generic_handler.py`
**Function**: `create_deployed_handler()`

Replace the silent None-to-empty-dict conversion with explicit rejection:

```python
def handler(job: Dict[str, Any]) -> Any:
    raw_input = job.get("input")
    if raw_input is None or (isinstance(raw_input, dict) and not raw_input):
        return {
            "success": False,
            "error": (
                "Empty or null input. RunPod serverless requires at least one "
                "field in the input dict. Use explicit values or pass null for "
                "optional parameters."
            ),
        }
    if not isinstance(raw_input, dict):
        return {
            "success": False,
            "error": f"Malformed input: expected dict, got {type(raw_input).__name__}",
        }
    job_input = raw_input
    # ... existing logic
```

**Error response shape**: `{"success": False, "error": ...}` -- consistent with existing error returns in this function (lines 245-248, 271-275).

### 4. Error Message Consistency

All three locations use the same core message with minor format differences to match their error response shape:

> Empty or null input. RunPod serverless requires at least one field in the input dict. Use explicit values or pass null for optional parameters, e.g. `{"input": {"param_name": null}}`.

The message is actionable: it tells the developer exactly what to do to fix the issue.

## Testing

### Unit Tests

1. **`call_with_body` validation** (new test file or existing `test_run_server_helpers.py`):
   - Empty Pydantic model (`model_fields_set == set()`) raises HTTPException 422
   - Pydantic model with explicit null (`model_fields_set == {"msg"}`) passes through
   - Pydantic model with values passes through
   - Plain dict body (no `model_fields_set`) passes through unchanged (backward compat)

2. **Handler template validation** (existing handler_generator tests):
   - `{"input": {}}` returns error dict
   - `{"input": null}` returns error dict
   - Missing `"input"` key returns error dict
   - `{"input": {"key": "value"}}` proceeds to function call

3. **`create_deployed_handler` validation** (existing generic_handler tests):
   - Same cases as handler template, verifying `{"success": False, "error": ...}` response shape

### No Integration Tests

All changes are input validation at function boundaries. Unit tests with mock Pydantic models and job dicts provide full coverage.
